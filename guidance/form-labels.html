--- title: "Guidance on Designing & Implementing Form Labels" ---

In an effort to improve the way in which Radancy includes form labels on our sites and products, this document seeks to introduce more clearly defined guidance that we can share with all team members. Especially designers, developers, and QA engineers. Inclusion of a proper label is critical and allows more people to understand and use the form. Omission or poor implementation of a label is also considered a direct failure of Success Criterion 3.3.2 Labels or Instructions. This guideline was intentionally omitted from the initial baseline release to give us enough time to educate and acclimate ourselves to the advice presented here, but its time has now come. We will begin to test against this guideline more vigorously, beginning in the 4th quarter of 2022.

The Rules

  1. All form fields must include a visible label. See exceptions.
  2. Placeholder values are not an appropriate alternative for labels.
  3. Placeholder values may not duplicate what is already found in a label.
  4. Placeholder values may only be used to provide supplemental or suggestive information.
  5. Critical information about a form field, must always be placed outside of the field.
  6. Placeholder values must always be contrast friendly.

Exceptions

  1. If static labels are not desired above the input field, the floating label technique may be used. See floating label demo.
  2. Certain types of forms are so ubiquitous that they have become a common and well understood convention across the web. For example, a global site search that utilizes a single input field and button may not require a visible label.
  3. Certain types of information may have labels omitted if they are grouped together properly. See grouping example.

Examples

The following examples are meant to explain and illustrate poor and best practice of labeling form fields.

Placeholder Used as Label (Avoid)

While using placeholders as labels may seem like a great way to reduce visual clutter, placeholders and labels are very different things. Please familiarize with the numerous pitfalls of using placeholder text to stand-in for proper labels.

Note: Placeholders should also not be used with accessibly hidden labels. While this may seem like a way to address accessibility, it is still prone to many of the same issues outlined below. Use the floating label pattern instead.

Placeholder Redundancy (Avoid)

Do not duplicate the label as placeholder text. Not only is this visually redundant, but assistive technology users may have to listen to it twice, too.

Proper Placeholder Use

Placeholder text should be used to provide supplemental information. For example, formatting or suggestive instruction is an ideal use of placeholder text.

Critical Information

Placeholder text does have its limitations. If a field is pre-populated with data or a user begins typing, the placeholder text will vanish (a problem for people who have cognitive disabilities, to name but one group). If formatted or supplemental information is critical, then instructions should be placed outside of a field, where it will always be present.

Your email address will not be shared with third-parties.

To convey this information to assistive technology users, use aria-describedby.

<label for="example-4-email-address">Email Address</label>
<input aria-describedby="email-message" id="example-4-email-address">
<p id="email-message"><em>Your email address will not be shared with third-parties.</em></p>

Contrast Friendly

Default placeholder text, often a lighter shade of gray, can be hard to read in some browsers, so always aim for a color of #767676 on white backgrounds.

Omitting Labels in Group

This is a code example of what one might do to group associated fields together without individual labels. For example, three separate fields for a single date entry. Here, a fieldset and legend (or equivalent ARIA) could be used to group them, along with aria-label for each input field.

<fieldset>
  <legend>Date (Month, Day, Year)</legend>
  <input aria-label="Month" placeholder="MM">
  <input aria-label="Day" placeholder="DD">
  <input aria-label="Year" placeholder="YY">
</fieldset>
